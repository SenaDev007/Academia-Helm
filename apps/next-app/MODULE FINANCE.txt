🔎 0️⃣ Dans ce contexte, “annulation” veut dire quoi ?

Dans un système financier institutionnel sérieux, annulation ≠ suppression.

❌ Ce que ce n’est PAS :

Supprimer une ligne de paiement

Effacer une erreur

Modifier rétroactivement un montant sans trace


✅ Ce que c’est :

> Une opération comptable inverse traçable.



Concrètement :

Si un paiement de 50 000 FCFA est enregistré par erreur :

On ne supprime pas la ligne

On crée une écriture inverse : -50 000 FCFA

On conserve :

utilisateur

date

motif

IP

device_id


ORION peut analyser la fréquence d’annulations

une banque

🔒 Empêcher suppression de transactions

CREATE OR REPLACE FUNCTION prevent_transaction_delete()
RETURNS trigger AS $$
BEGIN
  RAISE EXCEPTION 'Deletion of financial transactions is forbidden';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER no_delete_transaction
BEFORE DELETE ON "FinanceTransaction"
FOR EACH ROW
EXECUTE FUNCTION prevent_transaction_delete();


---

🔄 Mise à jour automatique du solde

CREATE OR REPLACE FUNCTION update_student_account_balance()
RETURNS trigger AS $$
BEGIN
  UPDATE "StudentAccount"
  SET totalPaid = (
    SELECT COALESCE(SUM(amount),0)
    FROM "FinanceTransaction"
    WHERE studentAccountId = NEW.studentAccountId
  ),
  balance = totalDue - totalPaid
  WHERE id = NEW.studentAccountId;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER update_balance_after_transaction
AFTER INSERT ON "FinanceTransaction"
FOR EACH ROW
EXECUTE FUNCTION update_student_account_balance();

Interdire modification transaction après clôture

CREATE OR REPLACE FUNCTION prevent_update_after_closure()
RETURNS trigger AS $$
BEGIN
  IF EXISTS (
    SELECT 1 FROM "CashClosure"
    WHERE tenantId = NEW.tenantId
      AND academicYearId = NEW.academicYearId
      AND date = DATE(NEW.createdAt)
  ) THEN
    RAISE EXCEPTION 'Transaction locked after closure';
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER lock_transaction_after_closure
BEFORE UPDATE ON "FinanceTransaction"
FOR EACH ROW
EXECUTE FUNCTION prevent_update_after_closure();

## Interdire modification transaction après clôture

```sql id="b9vpxe"
CREATE OR REPLACE FUNCTION prevent_update_after_closure()
RETURNS trigger AS $$
BEGIN
  IF EXISTS (
    SELECT 1 FROM "CashClosure"
    WHERE tenantId = NEW.tenantId
      AND academicYearId = NEW.academicYearId
      AND date = DATE(NEW.createdAt)
  ) THEN
    RAISE EXCEPTION 'Transaction locked after closure';
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER lock_transaction_after_closure
BEFORE UPDATE ON "FinanceTransaction"
FOR EACH ROW
EXECUTE FUNCTION prevent_update_after_closure();
```

```prisma id="7yfx8k"
model FinancialSettings {
  id                       String   @id @default(uuid())
  tenantId                 String
  academicYearId           String?

  blockingThreshold        Decimal  @default(0)
  allowPartialPayment      Boolean  @default(true)
  minimumInstallmentAmount Decimal?

  autoClosureEnabled       Boolean  @default(true)
  autoClosureTime          String   @default("23:59")

  reminderWarningDays      Int      @default(3)
  reminderUrgentDays       Int      @default(7)
  reminderFinalDays        Int      @default(15)

  cancellationDelayHours   Int      @default(24)

  budgetAlertThreshold     Int      @default(85)

  fedapayEnabled           Boolean  @default(false)
  fedapayPublicKey         String?
  fedapaySecretKey         String?

  createdAt                DateTime @default(now())
  updatedAt                DateTime @updatedAt
}
```

---

# 🔎 3️⃣ RÈGLES MÉTIER EXHAUSTIVES

---

## 🔐 A. Seuil de blocage

blockingThreshold = montant minimum déclenchant blocage.

Si balance > threshold → blocage automatique.

---

## 🔐 B. Paiement partiel

allowPartialPayment :

* true → autorisé
* false → paiement minimum tranche obligatoire

minimumInstallmentAmount :

* montant minimum accepté

---

## 🔐 C. Politique d’annulation

cancellationDelayHours :

* < délai → comptable
* > délai → directeur
* > clôture → promoteur uniquement

---

## 🔐 D. Paramétrage relances

Délais configurables :

* warningDays
* urgentDays
* finalDays

Empêche hardcoding.

---

## 🔐 E. Clôture automatique

autoClosureEnabled :

* true → cron actif
* false → manuel obligatoire

autoClosureTime :

* heure personnalisable

---

## 🔐 F. Budget

budgetAlertThreshold :

* % à partir duquel ORION déclenche alerte

---

## 🔐 G. Fedapay

Activation/désactivation
Clés sécurisées
Chiffrement obligatoire côté serveur

# 🧱 1️⃣ AUDIT AVANCÉ — NIVEAU INSTITUTIONNEL

## 🎯 Objectif

Aucune opération financière ne doit pouvoir être :

* modifiée sans trace
* supprimée
* antidatée
* manipulée après clôture

---

## 🗃 Schéma Prisma — Audit financier

```prisma
model FinancialAuditLog {
  id              String   @id @default(uuid())
  tenantId        String
  entityType      String
  entityId        String
  action          String
  oldValue        Json?
  newValue        Json?
  performedById   String
  performedAt     DateTime @default(now())
  ipAddress       String?
  deviceId        String?
}
```

---

## 🔒 Règles critiques

* Aucun DELETE réel → soft delete uniquement
* Modification transaction après clôture = interdit
* Toute modification crée audit automatique
* Hash interne de transaction (intégrité)

---

## 🔐 Hash d’intégrité transaction

Chaque transaction doit générer :

```
hash = SHA256(
  tenantId + studentId + amount + type + createdAt
)
```

Stocké dans FinanceTransaction.

Toute modification invalide le hash.

---

# 🤖 2️⃣ ANTI-FRAUDE IA — ORION FINANCIER

ORION ne doit pas être passif.

Il doit détecter :

* Multiples annulations en 24h
* Transactions nocturnes anormales
* Écart entre cash théorique & réel
* Clôture retardée répétée
* Comptable modifiant souvent mêmes élèves
* Paiements fractionnés suspects

---

## 🧠 Modèle détection anomalie

Créer table :

```prisma
model FinancialAnomaly {
  id            String   @id @default(uuid())
  tenantId      String
  type          String
  severity      String
  referenceId   String
  detectedAt    DateTime @default(now())
  resolved      Boolean  @default(false)
}
```

---

## 🧠 Types d’anomalies

* MULTIPLE_CANCELLATION
* ABNORMAL_TIME_TRANSACTION
* CASH_MISMATCH
* FREQUENT_ADJUSTMENT
* SUSPICIOUS_PARTIAL_PAYMENT
* MANUAL_OVERRIDE_EXCESS

---

ORION doit analyser :

* à chaque clôture
* à chaque annulation
* quotidiennement via cron

# 🔐 TRIGGERS SQL CRITIQUES

1️⃣ Empêcher modification après clôture :

```
IF EXISTS (SELECT 1 FROM CashClosure WHERE isClosed = true)
THEN RAISE EXCEPTION
```

2️⃣ Empêcher suppression transaction :

```
BEFORE DELETE → RAISE EXCEPTION
```

3️⃣ Vérifier hash intégrité :

Trigger BEFORE UPDATE :

Recalcul hash
Comparer
Si mismatch → bloquer
